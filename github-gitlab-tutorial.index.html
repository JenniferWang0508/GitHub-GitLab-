📖 GitHub & GitLab 完整教學指南

從零開始學習版本控制與協作開發平台 — 涵蓋每個頁面功能的詳盡教學

🌐平台介紹

什麼是 Git？

Git 是一套分散式版本控制系統（Distributed Version Control System），由 Linux 之父 Linus Torvalds 於 2005 年開發。它能追蹤檔案的每一次修改，讓多人協作開發時不會互相覆蓋程式碼。

💡 版本控制就像「無限次的 Ctrl+Z」— 你可以隨時回到任何一個歷史版本，比較差異，合併不同人的修改。

GitHub 是什麼？

🐙 基本資訊

  • 創立：2008 年，由 Tom Preston-Werner、Chris Wanstrath、PJ Hyett 創辦
  • 收購：2018 年被 Microsoft 以 75 億美元收購
  • 定位：全球最大的開源社群與程式碼託管平台
  • 用戶數：超過 1 億開發者
  • 網址：github.com

⭐ 核心特色

  • 全球最大的開源專案集中地
  • 強大的社群功能（Star、Fork、Follow）
  • GitHub Actions — 內建 CI/CD
  • GitHub Copilot — AI 程式碼助手
  • GitHub Pages — 免費靜態網站
  • GitHub Codespaces — 雲端開發環境

GitLab 是什麼？

🦊 基本資訊

  • 創立：2011 年，由 Dmitriy Zaporozhets 和 Valery Sizov 開發
  • 上市：2021 年 NASDAQ 上市（GTLB）
  • 定位：完整的 DevSecOps 平台
  • 版本：社群版（CE，開源免費）與企業版（EE）
  • 網址：gitlab.com 或自架

⭐ 核心特色

  • 完整 DevOps 生命週期整合（計畫→開發→測試→部署→監控）
  • 強大的內建 CI/CD（不需第三方）
  • 可自架（Self-hosted），企業最愛
  • 內建 Container Registry
  • 內建安全掃描（SAST/DAST）
  • Web IDE 線上編輯器

🔧Git 基礎指令

安裝 Git

作業系統安裝方式
Windows下載 git-scm.com 安裝包，或用 winget install Git.Git
macOSbrew install git 或安裝 Xcode Command Line Tools
Linux (Ubuntu)sudo apt install git

初始設定

# 設定使用者名稱與信箱（每台電腦只需設定一次）
git config --global user.name "你的名字"
git config --global user.email "your@email.com"

# 查看設定
git config --list

# 設定預設分支名稱為 main
git config --global init.defaultBranch main

# 設定預設編輯器（可選）
git config --global core.editor "code --wait"   # VS Code

核心指令速查表

📁 建立與複製倉庫 
# 在當前目錄初始化一個新的 Git 倉庫
git init

# 從遠端複製一個倉庫到本機
git clone https://github.com/user/repo.git

# 複製並指定資料夾名稱
git clone https://github.com/user/repo.git my-folder

# 只複製最新一次提交（淺克隆，節省時間）
git clone --depth 1 https://github.com/user/repo.git
📝 暫存與提交 
# 查看目前狀態（哪些檔案被修改、暫存）
git status

# 將檔案加入暫存區（Staging Area）
git add filename.txt        # 加入單一檔案
git add .                   # 加入所有變更
git add *.js                # 加入所有 .js 檔案
git add src/                # 加入整個目錄

# 提交暫存區的變更
git commit -m "描述這次修改做了什麼"

# 加入並提交（跳過 git add，只對已追蹤的檔案有效）
git commit -am "快速提交"

# 修改上一次提交的訊息
git commit --amend -m "新的提交訊息"
🌿 分支操作 
# 查看所有本地分支
git branch

# 查看所有分支（含遠端）
git branch -a

# 建立新分支
git branch feature/login

# 切換分支
git checkout feature/login
# 或（Git 2.23+）
git switch feature/login

# 建立並切換到新分支
git checkout -b feature/login
# 或
git switch -c feature/login

# 刪除已合併的分支
git branch -d feature/login

# 強制刪除未合併的分支
git branch -D feature/login

# 重新命名分支
git branch -m old-name new-name
🔀 合併與變基 
# 合併分支到當前分支
git merge feature/login

# 變基（Rebase）— 將當前分支的提交「接」到目標分支之後
git rebase main

# 互動式變基 — 可以修改、合併、刪除提交
git rebase -i HEAD~3   # 修改最近 3 個提交

# 解決合併衝突後繼續
git add .
git merge --continue
# 或
git rebase --continue

# 放棄合併/變基
git merge --abort
git rebase --abort
💡 Merge vs Rebase：Merge 保留完整歷史（有合併節點），Rebase 讓歷史更乾淨（線性）。在公共分支上避免 Rebase。
📡 遠端操作 
# 查看遠端倉庫
git remote -v

# 新增遠端倉庫
git remote add origin https://github.com/user/repo.git

# 推送到遠端
git push origin main

# 第一次推送並設定上游追蹤
git push -u origin main

# 從遠端拉取並合併
git pull origin main

# 只下載遠端資料但不合併
git fetch origin

# 刪除遠端分支
git push origin --delete feature/login
📋 查看歷史與差異 
# 查看提交歷史
git log

# 簡潔格式
git log --oneline

# 圖形化顯示分支
git log --oneline --graph --all

# 查看某檔案的歷史
git log -- filename.txt

# 查看工作目錄與暫存區的差異
git diff

# 查看暫存區與上次提交的差異
git diff --staged

# 查看兩個分支的差異
git diff main..feature/login
💾 暫存工作（Stash） 
# 暫存當前未提交的修改
git stash

# 加上描述
git stash save "做到一半的登入功能"

# 查看暫存列表
git stash list

# 恢復最近的暫存
git stash pop         # 恢復並刪除
git stash apply       # 恢復但保留

# 恢復指定的暫存
git stash pop stash@{2}

# 刪除暫存
git stash drop stash@{0}
git stash clear       # 清除所有
⏪ 撤銷與重置 
# 撤銷工作目錄的修改（回到上次提交的狀態）
git checkout -- filename.txt
# 或（Git 2.23+）
git restore filename.txt

# 取消暫存（從暫存區移出）
git reset HEAD filename.txt
# 或
git restore --staged filename.txt

# 軟重置 — 撤銷提交但保留修改在暫存區
git reset --soft HEAD~1

# 混合重置 — 撤銷提交，修改回到工作目錄
git reset --mixed HEAD~1

# 硬重置 — 完全撤銷，修改也會消失（危險！）
git reset --hard HEAD~1

# 建立一個「反向提交」來撤銷某次提交（安全）
git revert abc1234
⚠️ git reset --hard 會永久刪除未提交的修改！請確認後再使用。
📄 .gitignore 檔案

在專案根目錄建立 .gitignore 檔案，列出不需要追蹤的檔案：

# 編譯產出
/dist
/build
*.o
*.class

# 依賴目錄
node_modules/
vendor/
__pycache__/
venv/

# 環境變數與機密
.env
.env.local
*.pem
*.key

# IDE 設定
.vscode/
.idea/
*.swp
*.swo

# 系統檔案
.DS_Store
Thumbs.db

# 日誌
*.log
logs/
💡 github.com/github/gitignore 有各種語言和框架的 .gitignore 模板，可以直接使用。
🏷️ 標籤（Tag） 
# 建立輕量標籤
git tag v1.0.0

# 建立附註標籤（推薦）
git tag -a v1.0.0 -m "第一個正式版本"

# 查看所有標籤
git tag

# 推送標籤到遠端
git push origin v1.0.0
git push origin --tags    # 推送所有標籤

# 刪除標籤
git tag -d v1.0.0                    # 刪除本地
git push origin --delete v1.0.0      # 刪除遠端

建立帳號

1
前往 github.com/signup，輸入信箱、密碼、使用者名稱
2
完成信箱驗證與人機驗證
3
選擇方案 — Free 方案對個人開發者已經非常夠用

個人檔案設定（Profile）

點擊右上角頭像 → Your profile：

  • 頭像：上傳個人照片或識別圖片
  • Bio：簡短自我介紹
  • Company / Location / Website：補充資訊
  • README Profile：建立一個與你的使用者名稱同名的 Repository，裡面的 README.md 會顯示在你的個人頁面上。這是展示自己的好地方！

帳號設定（Settings）

點擊右上角頭像 → Settings：

🔑 SSH Keys 設定

SSH Key 讓你不用每次 push/pull 都輸入密碼：

# 1. 產生 SSH 金鑰
ssh-keygen -t ed25519 -C "your@email.com"

# 2. 查看公鑰
cat ~/.ssh/id_ed25519.pub

# 3. 複製公鑰內容

# 4. 到 GitHub → Settings → SSH and GPG keys → New SSH key
#    貼上公鑰，取個名稱（如 "My Laptop"）

# 5. 測試連線
ssh -T git@github.com
# 成功會顯示：Hi username! You've successfully authenticated
🔐 Personal Access Token (PAT)

PAT 用來取代密碼進行 HTTPS 驗證，或用於 API 存取：

1
Settings → Developer settings → Personal access tokens → Tokens (classic) 或 Fine-grained tokens
2
點擊 Generate new token
3
設定名稱、過期時間、勾選需要的權限（如 repo、workflow）
4
產生後立即複製！離開頁面後就看不到了
⚠️ Fine-grained tokens（新版）比 Classic tokens 更安全，可以限制只存取特定 Repo。建議優先使用。
🔔 Notifications 設定

Settings → Notifications：

  • Email notifications：選擇哪些事件要收到信件通知
  • Watching：預設 Watch 自己參與的 Repo，可改為 "Only participating" 減少通知量
  • GitHub Mobile：可啟用手機推播
  • Custom routing：不同組織的通知可以發到不同信箱
🎨 Appearance 外觀
  • Theme：淺色 / 深色 / 跟隨系統
  • Tab size：程式碼顯示的 Tab 寬度
  • Markdown editor：顯示偏好

Repository 的預設頁面，功能包括：

  • ⭐ Star：類似「收藏」，表示你喜歡或關注這個專案。Star 數量是衡量專案受歡迎程度的指標。

🐛GitHub — Issues

Issues 是 GitHub 的任務追蹤系統，用來回報 Bug、提出功能需求、討論技術問題。

建立 Issue

1
進入 Repo → Issues 分頁 → New issue
2
如果有 Issue Template，選擇適合的模板（如 Bug Report、Feature Request）
3
填寫標題和描述，支援 Markdown 格式
4
右側欄設定：Assignees（指派）、Labels（標籤）、Projects、Milestone

Issue 功能詳解

功能說明
Labels標籤分類，如 bug、enhancement、good first issue。可自訂顏色和名稱
Assignees指派負責人，最多 10 人
Milestone里程碑，將多個 Issue 歸類到同一個版本或目標，可追蹤進度百分比
Projects關聯到 Projects 看板
關聯 PR在 PR 描述中寫 Closes #123 或 Fixes #123，PR 合併時 Issue 會自動關閉
Reactions👍 👎 😄 🎉 😕 ❤️ 🚀 👀 — 用表情回應，減少無意義的 "+1" 留言
Lock鎖定 Issue 不允許新留言（維護者使用）
Pin將重要 Issue 釘選在列表頂部，最多 3 個

Issue Templates

在 Repo 中建立 .github/ISSUE_TEMPLATE/ 目錄，放入 YAML 或 Markdown 模板：

# .github/ISSUE_TEMPLATE/bug_report.yml
name: Bug Report
description: 回報一個 Bug
labels: ["bug"]
body:
  - type: textarea
    id: description
    attributes:
      label: 描述 Bug
      description: 清楚描述你遇到的問題
    validations:
      required: true
  - type: textarea
    id: steps
    attributes:
      label: 重現步驟
      description: 如何重現這個問題？
  - type: dropdown
    id: severity
    attributes:
      label: 嚴重程度
      options:
        - 低
        - 中
        - 高
        - 緊急

🔃GitHub — Pull Requests

Pull Request（PR）是 GitHub 的程式碼審查與合併機制。當你在分支上完成修改後，透過 PR 請求將修改合併到主分支。

建立 Pull Request

1
推送你的分支到遠端：git push origin feature/my-feature
2
到 GitHub Repo 頁面，會出現黃色提示條 "Compare & pull request"，點擊它
3
或者進入 Pull requests 分頁 → New pull request → 選擇 base 和 compare 分支
4
填寫標題和描述（建議說明修改了什麼、為什麼、如何測試）
5
設定 Reviewers、Assignees、Labels、Projects、Milestone
6
點擊 Create pull request

PR 功能詳解

功能說明
ConversationPR 主討論區，顯示所有留言、審查結果、CI 狀態
Commits列出 PR 中包含的所有提交
ChecksCI/CD 檢查結果（GitHub Actions 等）
Files changed顯示所有變更的差異（diff），可逐行留言
Reviewers指定審查者，他們可以 Approve / Request changes / Comment
Draft PR標記為草稿，表示還沒準備好被審查。點擊 "Ready for review" 即可轉為正式 PR
Auto-merge啟用後，當所有檢查通過且獲得批准，會自動合併

三種合併策略

Merge Commit

保留所有提交歷史，建立一個額外的合併提交。歷史最完整，但可能較雜亂。

git merge --no-ff feature

Squash and Merge

將所有提交壓縮成一個提交再合併。主分支歷史乾淨，但失去細節。

git merge --squash feature

Rebase and Merge

將提交逐一接到 base 分支之後，歷史線性且保留每個提交。

git rebase main && git merge --ff-only

如何選擇？

  • 個人專案 → Squash（簡潔）
  • 團隊開發 → Merge Commit（完整追蹤）
  • 開源專案 → 看專案規範

Code Review 技巧

  • 在 Files changed 頁面，點擊行號旁的 + 按鈕可以對特定行留言
  • 可以「建議修改」：留言中使用 ```suggestion 區塊，作者可以一鍵套用
  • 審查完成後點擊 Review changes：Comment / Approve / Request changes
  • 勾選 Viewed 標記檔案已審查完畢

⚡GitHub — Actions (CI/CD)

GitHub Actions 是內建的 CI/CD 平台，透過 YAML 設定檔自動化工作流程：建構、測試、部署。

核心概念

術語說明
Workflow一個自動化流程，定義在 .github/workflows/ 目錄下的 YAML 檔案
Event觸發 Workflow 的事件（push、pull_request、schedule 等）
JobWorkflow 中的一組步驟，在同一個 Runner 上執行
StepJob 中的單一步驟，可以是執行命令或使用 Action
Action可重用的程式單元，可從 Marketplace 取得或自行編寫
Runner執行 Workflow 的伺服器，GitHub 提供免費的 Ubuntu/Windows/macOS Runner
Secrets加密的環境變數，用於存放 API Key 等機密資訊

基本範例

# .github/workflows/ci.yml
name: CI Pipeline

on:
  push:
    branches: [ main, develop ]
  pull_request:
    branches: [ main ]

jobs:
  test:
    runs-on: ubuntu-latest

    strategy:
      matrix:
        node-version: [18, 20, 22]

    steps:
      # 1. 取出程式碼
      - uses: actions/checkout@v4

      # 2. 安裝 Node.js
      - uses: actions/setup-node@v4
        with:
          node-version: ${{ matrix.node-version }}
          cache: 'npm'

      # 3. 安裝依賴
      - run: npm ci

      # 4. 執行測試
      - run: npm test

      # 5. 執行 Lint
      - run: npm run lint

  deploy:
    needs: test           # 等 test job 通過才執行
    runs-on: ubuntu-latest
    if: github.ref == 'refs/heads/main'   # 只在 main 分支部署

    steps:
      - uses: actions/checkout@v4
      - run: npm ci
      - run: npm run build
      - name: Deploy
        env:
          DEPLOY_TOKEN: ${{ secrets.DEPLOY_TOKEN }}
        run: |
          echo "Deploying to production..."
          # 你的部署命令

🚀GitLab — CI/CD

GitLab CI/CD 是其最強大的功能之一，透過 .gitlab-ci.yml 定義自動化流水線。

核心概念

術語說明
Pipeline整個 CI/CD 流程，由多個 Stage 組成
Stage階段（如 build → test → deploy），同一 Stage 的 Job 平行執行
Job具體的工作項目，在 Runner 上執行
Runner執行 Job 的代理程式（GitLab 提供 Shared Runner，也可自建）
ArtifactsJob 產出的檔案，可在 Job 之間傳遞或下載
Variables環境變數，可在 Settings → CI/CD → Variables 中設定
Cache快取依賴目錄以加速建構
Environment部署目標環境（staging、production）

.gitlab-ci.yml 完整範例

# .gitlab-ci.yml

# 定義階段順序
stages:
  - build
  - test
  - deploy

# 全域設定
default:
  image: node:20-alpine
  cache:
    key: ${CI_COMMIT_REF_SLUG}
    paths:
      - node_modules/

# 全域變數
variables:
  NODE_ENV: test

# 安裝依賴（所有 Job 之前執行）
before_script:
  - npm ci

# ======== Build 階段 ========
build:
  stage: build
  script:
    - npm run build
  artifacts:
    paths:
      - dist/
    expire_in: 1 hour

# ======== Test 階段 ========
unit-test:
  stage: test
  script:
    - npm run test:unit
  coverage: '/Statements\s*:\s*(\d+\.?\d*)%/'

lint:
  stage: test
  script:
    - npm run lint

e2e-test:
  stage: test
  image: cypress/browsers:latest
  script:
    - npm run test:e2e
  allow_failure: true

# ======== Deploy 階段 ========
deploy-staging:
  stage: deploy
  script:
    - echo "Deploying to staging..."
    - ./scripts/deploy.sh staging
  environment:
    name: staging
    url: https://staging.example.com
  only:
    - develop
  when: manual

deploy-production:
  stage: deploy
  script:
    - echo "Deploying to production..."
    - ./scripts/deploy.sh production
  environment:
    name: production
    url: https://example.com
  only:
    - main
  when: manual
  allow_failure: false

🎯實戰範例 — 從零開始

範例：建立一個 Web 專案並多人協作

Step 1：建立 Repository

gh repo create my-web-app --public --clone
# 或
glab repo create my-web-app --public

cd my-web-app

Step 2：初始化專案

# 建立基本檔案
echo "# My Web App" > README.md
echo "node_modules/" > .gitignore

# 初始提交
git add .
git commit -m "Initial commit: add README and .gitignore"
git push -u origin main

Step 3：建立開發分支

# 建立 develop 分支作為開發主線
git checkout -b develop

# 建立功能分支

Step 4：開發功能

# 建立首頁

<!DOCTYPE html>
<html>

# 提交

# 其他開發者同步最新的 develop
git checkout develop
git pull origin develop

# 開始新功能
git checkout -b feature/add-about-page

❓常見問題 FAQ

Q: push 時出現 "Permission denied (publickey)" 怎麼辦？

SSH Key 沒設定好。依序檢查：

  1. 確認 SSH Key 是否存在：ls ~/.ssh/id_ed25519.pub
  2. 確認是否已加到 GitHub/GitLab：Settings → SSH Keys
  3. 測試連線：ssh -T git@github.com 或 ssh -T git@gitlab.com
  4. 如果用 HTTPS，改用 PAT 作為密碼
Q: merge 衝突怎麼解決？ 
# 1. Git 會在衝突檔案中標記衝突位置
<<<<<<< HEAD
你的修改
=======
別人的修改
>>>>>>> feature/other

# 2. 手動編輯檔案，保留需要的內容，刪除標記

# 3. 加入暫存並完成合併
git add conflicted-file.txt
git commit -m "Resolve merge conflict"
Q: 不小心把密碼/API Key commit 了怎麼辦？
⚠️ 重要：即使刪除檔案再提交，密碼仍然在 Git 歷史中！需要徹底清除。
  1. 立即撤銷該密碼/Key（在服務商處重新產生）
  2. 使用 git filter-repo 或 BFG Repo-Cleaner 清除歷史
  3. 強制推送：git push --force
  4. 通知所有協作者重新 clone
  5. 在 .gitignore 中加入機密檔案
# 使用 BFG Repo-Cleaner
bfg --delete-files .env my-repo.git
cd my-repo.git
git reflog expire --expire=now --all
git gc --prune=now --aggressive
git push --force
Q: GitHub 和 GitLab 該選哪個？
  • 選 GitHub：開源專案、個人作品集、重視社群曝光、使用 Copilot
  • 選 GitLab：企業內部使用、需要自架、重視 CI/CD、需要完整 DevOps 工具鏈
  • 兩者都用：很多開發者同時使用，GitHub 放開源，GitLab 放公司專案
Q: 如何撤銷已 push 的 commit？ 
# 方法 1：Revert（安全，建立反向提交）
git revert HEAD
git push

# 方法 2：Reset + Force Push（危險，會改寫歷史）
git reset --hard HEAD~1
git push --force

# ⚠️ 方法 2 在多人協作時可能造成問題！
# 建議只在個人分支上使用 force push
Q: fork 的 Repo 如何同步原專案的更新？ 
# 1. 新增原專案為 upstream remote
git remote add upstream https://github.com/original/repo.git

# 2. 取得最新更新
git fetch upstream

# 3. 合併到你的 main
git checkout main
git merge upstream/main
git push origin main

🏆最佳實踐

Commit 訊息規範

推薦使用 Conventional Commits 格式：

type(scope): description

# 常用 type：
feat:     新功能
fix:      修復 Bug
docs:     文件修改
style:    格式調整（不影響邏輯）
refactor: 重構（不是新功能也不是修 Bug）
test:     測試相關
chore:    建構、工具、依賴更新
ci:       CI/CD 設定

# 範例
feat(auth): add OAuth2 login support
fix(api): resolve null pointer in user endpoint
docs: update README with setup instructions
chore(deps): bump lodash from 4.17.20 to 4.17.21

分支策略

策略適合特點
Git Flow大型專案、有版本發布main + develop + feature + release + hotfix
GitHub Flow持續部署的專案main + feature branches，簡單直接
Trunk-based高頻部署的團隊所有人都在 main 上工作，搭配 Feature Flags

安全建議

  • ✅ 啟用二步驗證（2FA）
  • ✅ 使用 SSH Key 或 Fine-grained Token（不要用密碼）
  • ✅ 設定分支保護規則
  • ✅ 啟用 Dependabot / Dependency Scanning
  • ✅ 在 .gitignore 中排除機密檔案
  • ✅ 定期審查 Personal Access Tokens
  • ❌ 永遠不要把密碼、API Key 提交到 Git

高效使用技巧

  • 🔑 學會使用 SSH Key，避免每次輸入密碼
  • ⌨️ 熟記 Git 別名：git config --global alias.co checkout
  • 📝 寫好 README — 第一印象很重要
  • 🏷️ 善用 Labels 和 Milestones 管理進度
  • 🤖 設定 CI/CD 自動化重複性工作
  • 📊 定期查看 Insights/Analytics 了解專案健康度
  • 💬 善用 Issue/Discussion 進行非同步溝通
